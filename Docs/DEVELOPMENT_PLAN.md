# 開発プロダクト

**達成感を味わえるTODOリストアプリ**

## プロダクト発案理由

- 子どもが生まれてから勉強や個人開発する時間が減り、日々の達成感がなくネガティブになっている
- 勉強してなくても、「今日はこれをやった！えらい！」と思えるアプリが欲しくなった
- 「達成のためのTODOリストアプリ」は世の中にたくさんあるが、ゴール設定してストイックにTODO消化したいわけではなく、「今日はこれだけ作業した」「今月はこんなに作業した」と感じたいだけである。
- 意外と過去消化したTODOを見れるアプリが少なく(見れても過去30日までとか)、消化したTODOをいい感じに確認でき、達成感をより感じられるアプリを作りたいと思った

## プロダクト概要

###  1. 基本的なTODO機能

- 作成、編集、削除、完了、一覧表示

### 2. タグ作成機能

- TODOに追加できる情報の一つ
- 例：個人開発、家事、育児など

### 3. 難易度追加機能

- TODOに追加できる情報の一つ
- 大きいタスクと小さいタスクを同列で扱うと、大きいタスクをやればやるほどタスク消化数が減り達成感を味わいにくくなるため、大中小の3段階でタスクにレベル付けできるようにする

### 4.完了タスク閲覧機能

- 年月日別やタグ別で絞り込みできる
- 総完了件数および難易度ごとの完了件数を表示する

## 本プロダクトでやらないこと

- 達成したいゴールの設定(目的はゴールを達成することではなく、あくまで達成感を味わうことである。達成できなかった時のネガティブ感情も味わいたくない)
- 平均タスク消化数の表示(比較して少ないとモチベーションが下がる)

# 開発の進め方

## 概要

- いったん知見のあるLaravelで開発する
- フロントエンドは最初はbladeおよびバニラjsで開発し、後からReact に置き換える
- TDDおよびDDD → React → React Native の順で学んでいく

## 開発要件

### バックエンドとフロントエンドは二つに分ける

- React学習に最適：VercelはReact特化で高速ビルド、CDN、無料枠が充実。
- 明確な役割分担：LaravelはAPI、ReactはUIと分離し、現代的開発に即応。TDD/DDDに集中しやすい。
- React Native連携容易：Webとモバイルで同じAPIを再利用できる。
- コスト効率が良い：Vercelの無料枠でフロントエンド費用を抑え、HerokuはAPIのみで最小限に。
- 高い柔軟性：フロントとバックエンドを個別にスケール・技術変更できる。

### ローカル環境はバックエンド→Docker、フロントエンド→開発サーバ

#### バックエンド：ビルトインサーバーではなくDockerを使う理由

- 環境の一貫性: Dockerを使うと、開発環境（PHPのバージョン、拡張機能、Webサーバー、データベース）を本番環境（Herokuなど）とほぼ同じ構成にできます。これにより、「ローカルでは動くのにデプロイしたら動かない」といった問題を減らせます。ビルトインサーバーでは、このような環境の一貫性を保つのが難しいです。

- 環境構築の簡素化と管理: PHP、Webサーバー（Nginxなど）、データベース（PostgreSQL）といった複数のソフトウェアを、個別にローカルPCにインストールして設定するのは面倒で、他のプロジェクトとの競合も起こりえます。Dockerを使えば、これらの環境をコンテナとして隔離し、docker-compose upコマンド一つでまとめて起動・停止できます。

- CI/CDへの移行がスムーズ: DockerはCI/CDパイプラインの基盤となることが多く、ローカルでDockerを使う経験は、将来の自動テストや自動デプロイの設定に直結します。


#### フロントエンド：Dockerより開発サーバーを使う理由

- 高速な開発体験: Viteのような開発サーバーは、ホットリロードや高速なビルドのために高度に最適化されています。これらはローカルPC上で直接動かすことで、最高のパフォーマンスを発揮し、コード変更が即座にブラウザに反映されます。Dockerコンテナを介すると、一部のOSや設定でパフォーマンスが落ちる可能性があります。

- シンプルな管理: フロントエンドはNode.jsとnpm/Yarn、そして開発サーバーがあれば事足ります。バックエンドのように複数のサービス（Webサーバー、DBなど）を連携させる複雑さがないため、Dockerを導入するメリットが薄いです。

- デプロイ先との親和性: ReactをVercelにデプロイする場合、Vercelが内部で最適なNode.js環境を使ってビルドしてくれるため、ローカルでDockerを使ってコンテナ化する必要がありません。

### Docker関連ファイルはLaravelルートに置く

- コードと環境の同期: アプリコードとDocker設定が同じリポジトリで管理され、バージョンが一致する。
- シンプルな設定: docker-compose.yml内のパス指定が簡単になる。
- CI/CD連携の容易さ: 自動テストやデプロイ設定がスムーズ。
- 開発環境の再現性: git clone後すぐにdocker-compose upで環境を立ち上げられる。
