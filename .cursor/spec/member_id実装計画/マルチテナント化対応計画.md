# マルチテナント化対応計画

## 0. 概要

現状、ほぼ全てのデータテーブルに `member_id` カラムがなく、API 認証ミドルウェアもほぼ付いていないため、**ログインしたユーザーが他人のデータを閲覧・編集できてしまう** 状態となっている。

本ドキュメントでは、自分以外のメンバーもこのアプリを使える状態にするために、各データテーブルに `member_id` を導入し、ログイン中のメンバーのデータのみを取得・操作できるようにするための実装計画を定義する。

---

## 1. 現状分析

### 1-1. 認証

- `members` テーブルは既に存在（旧 `users` テーブルから移行済）
- `app/Models/Member.php` が `Authenticatable` + `HasApiTokens` を実装済み
- `AuthController::login()` は `Auth::guard('web')->login($member)` でセッション認証も併用しており、Sanctum SPA 認証パターンで動作可能
- 現状認証ミドルウェアが付与されているのは `/api/auth/me` と `/api/auth/logout` のみ
- `routes/web.php`（blade 画面）にも `auth` ミドルウェアが無いため、未ログインでも全画面 URL にアクセス可能

### 1-2. データ所有

- 全データテーブルに `member_id` カラムが存在しない
- 各リポジトリのクエリ（`findAll` / `findById` / `search` / `searchAll` / `save` 等）に `member_id` 絞り込みロジックが存在しない
- 一部コントローラ（例: `EarlyMaladaptiveSchemaController::show()`）は `Model::first()` で取得しており、グローバルスコープも無いため、誰のデータでも先頭 1 件を返してしまう

## 2. 採用方針サマリ

| 項目 | 採用方針 |
|------|---------|
| 実装方式 | **explicit（明示的） + B方式相当**：ユースケースで `auth()->id()` を取得し、リポジトリへ引数で渡す |
| メソッド命名 | **`findAllForMember(int $memberId)` / `searchForMember(...)` のように `ForMember` サフィックス**を付与し、memberId を必須引数とする |
| 認証ミドルウェア | **API（`auth:sanctum`）+ Web（`auth`）の両方**を付与し多層防御 |
| 既存データの member_id 埋め | **Seeder（`AssignMemberIdSeeder`）で手動実行**。member_id はハードコード |
| 管理画面（`/siteAdmPanel63/...`） | **今回はスコープ外**（既存の `HttpBasicAuth` のまま、別途対応） |
| マイグレーション分割 | **2 ファイル構成**：①全テーブルに `member_id` カラム追加（nullable）、②全テーブルを NOT NULL 化 |
| シングルレコード型テーブルの一意性 | **`member_id` に `UNIQUE` 制約を付与**（5 テーブル）。1 メンバー = 1 レコードを DB レベルで保証 |
| Route Model Binding 時の他人データ | **404 を返す**（`resolveRouteBinding` をオーバーライドして `where('member_id', auth()->id())` で絞り込み） |

## 4. 実装ステップ

実装は以下の順番で進める。各ステップ完了後に動作確認を行う。

> ⚠️ **Step 1 〜 Step 3 のコミット・プッシュ運用に関する重要事項**
>
> 本リポジトリの `Procfile` には `release: php artisan migrate --force` が設定されており、**Heroku への push 時の release phase で自動的にマイグレーションが実行される**。
>
> そのため Step 1（カラム追加）と Step 3（NOT NULL 化）のマイグレーションを **同一コミットに含めて push してしまうと、Step 2（Seeder 手動実行）を挟めず、Step 3 の NOT NULL 化が空の `member_id` カラムに対して走り、即エラーで本番が落ちる**。
>
> したがって **Step 1 / Step 2 / Step 3 はそれぞれ独立したコミット・独立した push** で進めること（ローカル環境でも本番同等の運用を徹底し、事故を防ぐ）。

### Step 1: マイグレーション①（カラム追加）

各テーブルに `member_id` カラムを **nullable** で追加する。

- ファイル名: `database/migrations/2026_05_03_000001_add_member_id_to_user_data_tables.php`
- 全 15 テーブルを 1 ファイルでまとめて処理（DDL のみで副作用が少ないため）
- 外部キー制約: `members.id` を参照、`cascadeOnDelete()`
- 配置位置: `id` の直後（`after('id')`）
- Index 種別:
  - **マルチレコード型 10 テーブル**: 検索性能のため `member_id` 単独の通常 `index` を貼る
  - **シングルレコード型 5 テーブル**（3-1-a 参照）: `member_id` に **`UNIQUE` 制約**を貼る（nullable な UNIQUE は MySQL / MariaDB / PostgreSQL いずれも複数 NULL を許容するため Step 2 までは安全に運用可能）

#### 1-a. コミット・プッシュ運用

- **このマイグレーションファイルのみを 1 つのコミットに含める**（Step 3 のファイルは絶対に同梱しない）
- コミットメッセージ例: `feat(db): add nullable member_id column to user data tables (step1/3)`
- push すると Heroku の release phase で `php artisan migrate --force` が自動実行され、本番 DB にカラムが追加される
- **push 後、Heroku のビルドログで release phase が success（exit code 0）であることを必ず確認**してから Step 2 へ進む

> シングルレコード型テーブルで Step 2 完了後に二重レコードが残ると Step 3 の NOT NULL 化で UNIQUE 違反となる。Step 2 の Seeder で「最新 1 件以外を削除」する処理を必ず通すこと（詳細は Step 2 を参照）。

### Step 2: Seeder 作成・実行（既存データの埋め）

`database/seeders/AssignMemberIdSeeder.php` を新規作成。

- member_id は **ハードコード**（例: `private const TARGET_MEMBER_ID = 4`）
- **マルチレコード型 10 テーブル**: `DB::table('xxx')->whereNull('member_id')->update(['member_id' => self::TARGET_MEMBER_ID]);` を実施
- **シングルレコード型 5 テーブル**（3-1-a 参照）: 以下の手順で処理する
  1. `id` 降順で最新 1 件のみ取得
  2. 最新 1 件以外を `delete` で物理削除（UNIQUE 違反回避のため）
  3. 残った 1 件に `member_id = TARGET_MEMBER_ID` を設定
  4. レコードがそもそも 0 件の場合はスキップ（後続のユーザー操作で初回作成される）
- 環境ごとに ID を変えたい場合は、コミット前に Seeder ファイルを直接編集して対応

#### 2-a. コミット・プッシュ運用

- **この Seeder ファイルのみを 1 つのコミットに含める**（Step 3 のマイグレーションは絶対に同梱しない）
- コミットメッセージ例: `chore(seed): add AssignMemberIdSeeder for existing data (step2/3)`
- push すると Heroku の release phase で `php artisan migrate --force` が走るが、新規マイグレーションが無いので何もせず終わる（Seeder は自動実行されない）
- push 後、本番に Seeder ファイルが配置されたことを確認

#### 2-b. Seeder の手動実行（各環境）

`php artisan db:seed --class=AssignMemberIdSeeder` で **環境ごとに手動実行**する。

- ローカル: `php artisan db:seed --class=AssignMemberIdSeeder`
- 本番（Heroku）: `heroku run php artisan db:seed --class=AssignMemberIdSeeder --app <app-name>`
- レビューアプリ: 必要に応じて同様に手動実行

実行前のチェックリスト:

- [ ] DB バックアップ取得済み
- [ ] `TARGET_MEMBER_ID` のハードコード値が当該環境の本人 `members.id` と一致している
- [ ] シングルレコード型 5 テーブルの `count(*)` を取得し、削除対象件数を把握済み
- [ ] 特に `happy_schema_action_plans` は現行 `latest()->first()` 運用のため複数レコード残存の可能性が高い → 必ず件数確認

実行後のチェックリスト:

- [ ] 全 15 テーブルで `WHERE member_id IS NULL` となるレコードが 0 件
- [ ] シングルレコード型 5 テーブルで `count(*) <= 1`

> ⚠️ Seeder 実行 **完了** を確認してから Step 3 へ進む。未実行のまま Step 3 を push すると `member_id IS NULL` の行があるテーブルで NOT NULL 制約違反となり、release phase が失敗して本番が起動できなくなる。

### Step 3: マイグレーション②（NOT NULL 化）

- ファイル名: `database/migrations/2026_05_03_000002_make_member_id_not_null_on_user_data_tables.php`
- 全 15 テーブルの `member_id` を NOT NULL に変更
- このマイグレーション実行前に **必ず Step 2 の Seeder が（push 対象の全環境で）実行済みであること**

#### 3-a. コミット・プッシュ運用

- **このマイグレーションファイルのみを 1 つのコミットに含める**（Step 4 以降のアプリケーションコードは絶対に同梱しない）
- コミットメッセージ例: `feat(db): make member_id not null on user data tables (step3/3)`
- push 前に **全対象環境で Step 2 の Seeder 実行が完了していること** を再確認する
- push すると Heroku の release phase で `php artisan migrate --force` が自動実行され、本番 DB の `member_id` が NOT NULL 化される
- **push 後、Heroku のビルドログで release phase が success（exit code 0）であることを必ず確認**してから Step 4 へ進む

> Step 1 → Step 2 → Step 3 を **本番含む各環境ごとに実施** すること。順序を守らないと release phase で migrate が失敗し、本番アプリが起動できなくなる。

### Step 4: 認証ミドルウェア追加

#### 4-1. `routes/api.php`

- `auth/login` / `auth/register` / `auth/email/verify` / `auth/email/resend` 以外の全 API エンドポイントを `auth:sanctum` ミドルウェアで囲む

#### 4-2. `routes/web.php`

- 認証関連ページ（`/`、`/login`、`/register`、`/verify-email`）以外の全画面ルートを `auth` ミドルウェアで囲む
- 管理画面（`/siteAdmPanel63/...`）はスコープ外（既存 HttpBasicAuth のまま）

#### 4-3. `bootstrap/app.php`

- 未認証時のリダイレクト先を `/login` に設定（`Authenticate::redirectUsing()` ないし Middleware Configuration の `redirectGuestsTo`）

### Step 5: モデル修正（15 モデル）

各 Eloquent モデル（`app/Infrastructure/Database/Models/`）に以下を実施:

- `$fillable` に `'member_id'` を追加
- `member()` リレーション（`belongsTo(Member::class)`）を追加
- `resolveRouteBinding()` をオーバーライドし、`where('member_id', Auth::id())` で絞り込み（他人のデータは自動的に 404）

### Step 6: リポジトリインタフェース修正（Domain 層）

`app/Domain/Repository/*RepositoryInterface.php` の各メソッドに `int $memberId` 引数を追加。
メソッド名は **`ForMember` サフィックス** を付ける（管理画面用と取り違えないよう、意図を明示）。

例:

```php
interface ColumnRepositoryInterface
{
    public function findAllForMember(int $memberId): array;
    public function findByIdForMember(int $id, int $memberId): ?ColumnEntity;
    public function searchForMember(SearchCriteriaData $criteria, array $searchableColumns, int $memberId): array;
    public function searchAllForMember(SearchCriteriaData $criteria, array $searchableColumns, int $memberId): array;
    public function saveForMember(ColumnEntity $column, int $memberId): ColumnEntity;
    public function saveWithTagsForMember(ColumnEntity $column, array $tagIds, int $memberId): array;
    public function deleteForMember(int $id, int $memberId): void;
}
```

> 将来の管理画面用は `findAllForAdmin()` 等を**別メソッドとして**追加する想定（今回は実装しない）。

### Step 7: リポジトリ実装修正（Infrastructure 層）

`app/Infrastructure/Repository/Eloquent*Repository.php` の各メソッドで以下を実施:

- 全クエリに `->where('member_id', $memberId)` を追加
- 新規 `save` 時は `$model->member_id = $memberId;` を明示的に設定
- 更新 `save` 時は `findOrFail()` の前段に `where('member_id', $memberId)` を入れて、他人のデータを更新できないようにする

#### 7-a. シングルレコード型リポジトリの `findFirst()` 統一

`EloquentHappySchemaActionPlanRepository::findFirst()` のみ現状 `HappySchemaActionPlanModel::latest()->first()` で複数レコード時の保護策が入っているが、`UNIQUE(member_id)` 制約導入後は最大 1 件しか存在し得ないため、他 4 テーブル（`early_maladaptive_schemas` / `safe_places` / `mode_maps` / `healthy_adult_mode_images`）と揃えて `where('member_id', $memberId)->first()` に統一する。

対象 5 リポジトリ:

- `EloquentEarlyMaladaptiveSchemaRepository`
- `EloquentSafePlaceRepository`
- `EloquentModeMapRepository`
- `EloquentHappySchemaActionPlanRepository`（`latest()` 削除）
- `EloquentHealthyAdultModeImageRepository`

### Step 8: ユースケース修正（Application 層）

`app/Application/UseCase/**/*.php` の各ユースケースで以下を実施:

- `Auth::id()` を取得（または `Auth::id() ?? throw` でガード）
- リポジトリ呼び出し時に `int $memberId` を引数で渡す

例:

```php
class CreateColumnUseCase
{
    public function __construct(
        private ColumnRepositoryInterface $columnRepository,
    ) {}

    public function handle(ColumnData $data): array
    {
        $memberId = Auth::id();
        if ($memberId === null) {
            throw new \RuntimeException('未認証状態でユースケースが呼ばれた');
        }

        $column = ColumnEntity::create(/* ... */);

        return $this->columnRepository->saveWithTagsForMember($column, $data->tagIds, $memberId);
    }
}
```

### Step 9: コントローラ修正

#### 9-1. 全件取得系（`first()` 等）の絞り込み

- `EarlyMaladaptiveSchemaController::show()` の `EarlyMaladaptiveSchema::first()` を、`Auth::id()` の memberId で絞り込むように修正
- 同様に `SafePlace`、`ModeMap`、`HappySchemaActionPlan`、`HealthyAdultModeImage` の `show()` も要確認

#### 9-2. 子テーブル（`ProblemSolvingSolution` / `ProblemSolvingPlan`）の親所有者チェック

- `ProblemSolvingController::addSolution / updateSolution / deleteSolution / addPlan / updatePlan / deletePlan` 等のメソッドで、子モデルの `problem_solving_id` が引数の `$problemSolving->id` と一致するかチェック
- 不一致なら `abort(404)`

例:

```php
public function updateSolution(
    UpdateSolutionRequest $request,
    ProblemSolving $problemSolving,
    ProblemSolvingSolution $solution,
    UpdateSolutionUseCase $updateSolution,
): JsonResponse {
    if ($solution->problem_solving_id !== $problemSolving->id) {
        abort(404);
    }
    // ...既存ロジック
}
```

#### 9-3. `ProblemSolvingController::hasOverdueReflection()`

- `ProblemSolvingPlan::whereNotNull(...)` のクエリにも `whereHas('problemSolving', fn($q) => $q->where('member_id', Auth::id()))` を追加

### Step 10: 動作確認

- ログイン状態で他人の ID（DB に存在するが現メンバーが所有していない ID）を直接 URL で叩き、404 が返るか確認

### 6-1. マイグレーション①（カラム追加）

```php
return new class extends Migration
{
    /** マルチレコード型（通常 index） */
    private const MULTI_RECORD_TABLES = [
        'copings',
        'columns',
        'writing_disclosures',
        'problem_solvings',
        'simple_notepads',
        'stressor_and_responses',
        'support_networks',
        'schema_mode_monitorings',
        'dialogue_works',
        'chronologies',
    ];
```

### 6-2. Seeder

```php
class AssignMemberIdSeeder extends Seeder
{
    private const TARGET_MEMBER_ID = 4;

    /** マルチレコード型（全件に member_id を埋める） */
    private const MULTI_RECORD_TABLES = [
        'copings', 'columns', 'writing_disclosures', 'problem_solvings',
        'simple_notepads', 'stressor_and_responses', 'support_networks',
        'schema_mode_monitorings', 'dialogue_works', 'chronologies',
    ];

    /** シングルレコード型（最新 1 件のみ残して member_id を埋める） */
    private const SINGLE_RECORD_TABLES = [
        'early_maladaptive_schemas',
        'safe_places',
        'mode_maps',
        'happy_schema_action_plans',
        'healthy_adult_mode_images',
    ];

    public function run(): void
    {
        if (! DB::table('members')->where('id', self::TARGET_MEMBER_ID)->exists()) {
            throw new \RuntimeException(
                'TARGET_MEMBER_ID=' . self::TARGET_MEMBER_ID . ' が members に存在しません。Seeder のハードコード値を確認してください。'
            );
        }

        // マルチレコード型: 全件に member_id を埋める
        foreach (self::MULTI_RECORD_TABLES as $table) {
            $count = DB::table($table)
                ->whereNull('member_id')
                ->update(['member_id' => self::TARGET_MEMBER_ID]);
            $this->command->info(sprintf('[%s] %d 件を member_id=%d に更新', $table, $count, self::TARGET_MEMBER_ID));
        }

        // シングルレコード型: 最新 1 件のみ残して、それ以外を削除してから member_id を埋める
        foreach (self::SINGLE_RECORD_TABLES as $table) {
            $totalCount = DB::table($table)->count();

            if ($totalCount === 0) {
                $this->command->info(sprintf('[%s] レコード 0 件、スキップ', $table));
                continue;
            }

            $latestId = DB::table($table)->orderByDesc('id')->value('id');

            // 最新 1 件以外を物理削除（UNIQUE 違反防止）
            $deletedCount = DB::table($table)->where('id', '!=', $latestId)->delete();

            // 残った最新 1 件に member_id を設定
            $updatedCount = DB::table($table)
                ->where('id', $latestId)
                ->whereNull('member_id')
                ->update(['member_id' => self::TARGET_MEMBER_ID]);

            $this->command->info(sprintf(
                '[%s] %d 件削除、最新 id=%d を member_id=%d に更新（更新件数: %d）',
                $table, $deletedCount, $latestId, self::TARGET_MEMBER_ID, $updatedCount
            ));
        }
    }
}
```

## 7. 注意事項・リスク

### 7-1. データ移行時の事故防止

- **本番環境でも Seeder は手動実行**。自動実行に組み込まないこと
- Seeder のハードコード値（`TARGET_MEMBER_ID`）は環境ごとに事前に確認・編集する
- Seeder 実行前に必ず DB バックアップを取得する
- マイグレーション②（NOT NULL 化）の前に必ず Seeder が完了していること
- **シングルレコード型 5 テーブルでは Seeder が「最新 1 件以外を物理削除」する**ため、複数レコードが残存している場合の影響を事前確認すること（特に `happy_schema_action_plans` は `latest()->first()` 運用のため複数レコードがあり得る）。Seeder 実行前に各テーブルの `count(*)` を取得し、削除対象件数を必ず把握する

### 7-2. デプロイ時の手順（コミット単位）

本リポジトリの `Procfile` は以下のように release phase で migrate を自動実行する:

```
release: php artisan migrate --force
web: vendor/bin/heroku-php-apache2 public/
```

そのため、Step 1 / Step 2 / Step 3 のマイグレーション・Seeder は **必ず別コミット・別 push** で進めること。1 コミットにまとめて push すると Step 2 の手動 Seeder を挟めず、release phase の migrate が失敗し本番が起動不能になる。

| コミット | 含めるファイル | 本番 release phase の挙動 | デプロイ後の手動作業 |
|---------|---------------|------------------------|---------------------|
| **Commit 1（Step 1）** | `2026_05_03_..._add_member_id_to_user_data_tables.php` のみ | 自動 migrate → カラム追加 | DB バックアップ取得・release 成功確認 |
| **Commit 2（Step 2）** | `database/seeders/AssignMemberIdSeeder.php` のみ | 自動 migrate（変更なしで通過） | `heroku run php artisan db:seed --class=AssignMemberIdSeeder --app <app>` を**手動実行**し、`member_id IS NULL` が全テーブル 0 件であることを確認 |
| **Commit 3（Step 3）** | `2026_05_03_..._make_member_id_not_null_on_user_data_tables.php` のみ | 自動 migrate → NOT NULL 化 | release 成功確認 |
| **Commit 4 以降（Step 4〜9）** | アプリケーションコード（モデル・リポジトリ・ユースケース・コントローラ・ルーティング） | 自動 migrate（変更なしで通過） | 動作確認（Step 10） |

#### デプロイフロー（チェックリスト）

1. デプロイ前に **DB バックアップ取得**（Heroku の場合 `heroku pg:backups:capture --app <app>`）
2. **Commit 1（Step 1）** を push → release phase 成功を確認
3. **Commit 2（Step 2）** を push → release phase 成功を確認
4. `heroku run php artisan db:seed --class=AssignMemberIdSeeder --app <app>` を **手動実行**
5. `heroku pg:psql --app <app>` 等で `SELECT COUNT(*) FROM <each_table> WHERE member_id IS NULL;` を全 15 テーブル分確認 → 0 件であること
6. **Commit 3（Step 3）** を push → release phase 成功を確認
7. **Commit 4 以降（Step 4〜9）** を push（複数コミットに分割しても可）
8. **Step 10 の動作確認**を本番環境で実施

> ⚠️ Step 1 / Step 2 / Step 3 を**同一プルリクに含めて squash merge** すると、本番の release phase で 1 コミットとして扱われてしまう（Step 1 と Step 3 のマイグレーションが同時に走る）。**マージ戦略は必ず `Rebase and merge` または `Create a merge commit` を選び、コミット履歴を分けたまま本番に到達させること**。

### 7-3. 既存セッション

- 既存のログイン中セッションは Step 1〜3 完了後もそのまま使える想定
- `member_id` で絞られるため、Seeder で割り当てた `TARGET_MEMBER_ID` のメンバーでログインしている場合、自動的に自分のデータだけ見える状態になる

### 7-4. グローバルスコープ未採用の理由

- explicit 方式では Eloquent Global Scope を使わない
- 理由: 「未認証時に空クエリで動く」ような暗黙挙動を避け、引数で `memberId` を明示することで意図を明確化
- 代わりに認証ミドルウェアで未認証アクセスを完全にブロックする多層防御

### 7-5. 子テーブルのアクセス制御

- `ProblemSolvingSolution` / `ProblemSolvingPlan` には `member_id` を持たせない
- 親（`ProblemSolving`）の `resolveRouteBinding` で他人のものは 404 になる
- ただし子の ID 単独でアクセスされる場合に備え、コントローラで `$solution->problem_solving_id !== $problemSolving->id` チェックを入れる

### 7-6. 管理画面（admin）

- `/siteAdmPanel63/...` 配下のルートは今回の変更対象外
- 既存の `HttpBasicAuth` ミドルウェアで保護され続ける
- 将来、管理者ロールを導入する場合は別途対応する
- 管理画面用に `findAllForAdmin()` 等のリポジトリメソッドを追加する想定（今回は未実装）
